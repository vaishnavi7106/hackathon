Pillar 5
Real-Time Pest & Disease Outbreak Alert
Individual farmers can only see disease on their own land. By the time a visible outbreak spreads across a district, the window for preventive action has closed. Pillar 5 turns every Crop Sentinel diagnosis into a collective early-warning signal — detecting disease clusters across farms in real time, alerting farmers before symptoms appear on their land, and escalating critical outbreaks directly to the Tamil Nadu Agriculture Department.
Problem Being Solved
No early warning system: disease outbreaks spread farm to farm with no coordinated alert — farmers learn from neighbours days or weeks after the optimal treatment window
Individual blind spot: a single farmer's diagnosis is just data; 10 diagnoses within 10 km in 7 days is an outbreak — but no existing tool connects these dots
Government detection lag: TN Agriculture Department identifies district-level outbreaks 3–4 weeks after onset through manual field surveys — far too slow for intervention
Preventive treatment is 5x cheaper than curative: a farmer who sprays neem oil 5 days before symptoms appear saves 80% of the crop loss vs. one who waits to see visible damage
Estimated avoidable crop loss from late outbreak response: 10–15% of annual yield for affected crops
System Architecture
Pillar 5 is built entirely on top of existing Pillar 1 infrastructure — no new data collection, no new ML model. Every Crop Sentinel diagnosis already runs; Pillar 5 geo-tags those results and adds a detection layer on top.

Database Schema
Two new tables extend the existing PostgreSQL database. One change to the /diagnose endpoint — insert a geo-tagged row after every inference — is the only modification to existing Pillars.
Table
Key Fields
Purpose
disease_reports
farmer_id, disease_class, confidence, location (PostGIS POINT), reported_at, crop_type, is_confirmed
Stores every Crop Sentinel result with GPS coordinates. Confidence threshold >70% required for outbreak eligibility. is_confirmed flag lets TN Agri Dept validate or reject reports.
outbreak_alerts
disease_class, center_location, radius_km, report_count, severity, detected_at, alert_sent, affected_reports[]
One row per detected outbreak cluster. severity computed from report_count (LOW: 3+, MEDIUM: 6+, HIGH: 10+, CRITICAL: 20+). affected_reports stores UUID array of contributing disease_report IDs.
A PostGIS spatial index on disease_reports.location enables sub-10ms radius queries even at 100,000+ rows — critical for the detection engine to run quickly across all active disease types simultaneously.
Outbreak Detection Engine — The Key Innovation
The detection engine runs as a background cron job every 6 hours via APScheduler, embedded in the existing FastAPI application — no separate service required. For each disease type reported in the last 7 days, it uses PostGIS ST_DWithin to count how many reports of the same disease fall within 10 km of each anchor point. When 3 or more reports cluster within that radius and time window, an outbreak is declared.
Parameter
Value & Rationale
Cluster threshold
3 reports minimum — below this, coincidence is more likely than an outbreak. Adjustable per disease type in config.
Radius
10 km — matches typical spore dispersal range for fungal diseases and insect pest flight range for most Tamil Nadu crops.
Lookback window
7 days — captures fast-moving outbreaks while filtering old resolved infections from triggering false re-alerts.
Confidence filter
Only reports with >70% model confidence enter the clustering algorithm. Low-confidence diagnoses are stored but excluded from outbreak detection.
Deduplication
Before creating a new outbreak, the engine checks if an alert for the same disease already exists within 10 km in the last 24 hours — prevents duplicate notifications for the same event.
Run frequency
Every 6 hours — balances detection speed against database query load. Can be reduced to 1 hour with Redis caching layer.

Push Notification System
Firebase Cloud Messaging (FCM) is used for push delivery — free tier is sufficient for hackathon scale, works natively with React PWA via Service Workers, and supports Android, iOS, and desktop browsers with a single integration. FCM tokens are collected once during farmer onboarding and stored in the farmers table.
Farmers receive alerts only if both conditions are true: (1) they are within 10 km of the outbreak centroid, AND (2) they grow the affected crop type. A rice disease outbreak does not alert sugarcane farmers in the same area — this precision prevents alert fatigue.

Tamil Nadu Agriculture Department Integration
For HIGH and CRITICAL outbreaks only, Uzhavar AI fires a webhook to the TN Agriculture Department's reporting system. The payload includes disease type, GPS centroid, affected district, farmer count, severity, and a direct link to the Uzhavar AI outbreak dashboard for that event. Non-blocking implementation — if the government API is unavailable, the failure is logged silently without affecting the farmer-facing alert flow.
For the hackathon demo: if the live TN Agri API endpoint is unavailable, a mock webhook log in the admin dashboard shows a real-time scrolling feed — "✅ Outbreak report dispatched to TN Agriculture Department, Thanjavur District — July 10, 2025, 06:00 AM" — which communicates the integration story clearly to judges without requiring a live government API connection.
In-App Alert Banner
When a farmer opens Uzhavar AI and an active outbreak exists within 10 km of their registered location, a full-width alert banner appears at the top of the home screen before any other content. The banner is colour-coded by severity, shows the disease name in Tamil, the number of affected farmers, and contains a single call-to-action button that opens the Crop Sentinel camera directly. Farmers who dismiss the banner can retrieve it from a Notifications tab.

Data Sources
Crop Sentinel diagnoses (Pillar 1) — primary data source; every diagnosis from every farmer feeds this system automatically with zero additional farmer effort
Farmer GPS location — collected once at onboarding (district + village); refined with device GPS on each app open for accurate radius matching
PostGIS / PostgreSQL — spatial clustering engine; ST_DWithin enables radius queries in milliseconds at scale
Firebase Cloud Messaging (FCM) — push notification delivery; free tier supports up to 1 million messages/month
IMD weather data (Pillar 2) — cross-referenced to flag whether outbreak conditions correlate with recent rainfall or humidity spikes, adding agronomic context to alerts
Tamil Nadu Agriculture Department disease calendars — used to set seasonal sensitivity thresholds (lower cluster threshold during peak disease seasons)
Implementation Steps
Add PostGIS extension to existing PostgreSQL instance — single SQL command; no infrastructure change required
Create disease_reports and outbreak_alerts tables with spatial indexes — schema append, does not modify existing tables
Add geo-tag insert to /diagnose endpoint — one additional database write after existing ML inference; adds <5ms latency
Build outbreak_engine.py — APScheduler cron, ST_DWithin clustering logic, severity computation, deduplication check
Register FCM in Firebase Console — generate VAPID key; add Service Worker registration to React PWA (one-time setup, ~2 hours)
Build send_outbreak_alerts() function — FCM MulticastMessage, crop-type filter, Tamil notification text, mark alert_sent in DB
Build /api/outbreaks/active GET endpoint — returns active outbreak records with lat/lng for map rendering
Integrate Leaflet.js map in React PWA — OutbreakMap component, Circle layers, colour-coded by severity, 5-minute polling
Build OutbreakBanner component — home screen alert, severity colour, Tamil text, deep link to Crop Sentinel camera
Implement TN Agri Dept webhook — httpx POST, non-blocking, HIGH/CRITICAL filter, mock log for demo fallback
Add Twilio SMS for CRITICAL severity — single API call, district Agricultural Officer phone number from config
