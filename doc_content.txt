Pillar 1
Crop Sentinel

Crop diseases are the single largest controllable cause of yield loss for small Tamil Nadu farmers. By the time a farmer can see disease symptoms, 30–40% of the crop is already affected. Crop Sentinel uses computer vision to identify diseases from a smartphone photo and delivers actionable treatment guidance — including both modern and traditional indigenous remedies — within seconds.

Problem Being Solved
Late detection: farmers identify disease 2–3 weeks after optimal treatment window
Misdiagnosis: incorrect identification leads to wrong pesticide application, wasting money
No treatment alternatives: farmers unaware of cost-effective indigenous remedies (neem oil, compost teas)
Estimated crop loss attributable to late/wrong disease response: 25–30% of annual yield

ML Model: MobileNetV2 with Transfer Learning
Why MobileNetV2
Architecture
Convolutional Neural Network — MobileNetV2 (inverted residuals + depthwise separable convolutions)
Why this model
Designed for mobile inference — small footprint, fast, accurate on image classification tasks
Pre-training
ImageNet weights (transfer learning) — fine-tuned on PlantVillage disease dataset
Input
RGB photo of crop leaf, 224x224 pixels (captured via PWA camera or uploaded)
Output
Disease class + confidence score + SHAP explanation map
Model size
~14 MB — small enough to run on-device for offline use
Inference time
< 300ms on mid-range smartphone

Training Data
Primary dataset
PlantVillage — 54,306 images, 26 diseases across 14 crop species
Augmentation
Rotation, brightness shift, horizontal flip — to handle real field photo variation
Tamil Nadu crops
Prioritise: rice, sugarcane, banana, groundnut, cotton, tomato
Farmer-contributed
In-app photo submission feeds a growing Tamil Nadu-specific dataset over time
Validation split
80% train / 10% validation / 10% test — report accuracy on test set for judges

SHAP Explainability — The Key Innovation
Most disease detection apps return a label and a number. Farmers don't trust black boxes. SHAP (SHapley Additive exPlanations) generates a heatmap overlaid on the farmer's photo showing exactly which leaf region triggered the diagnosis. This serves two purposes: it builds farmer trust, and it flags false positives where the model is focusing on soil or background instead of the leaf.
Library
SHAP GradientExplainer — efficient for CNN models
Output format
Heatmap overlay rendered directly in the PWA on the original photo
Tamil label
"இந்த இலையின் மஞ்சள் பகுதி நோயை காட்டுகிறது" (This yellow area on the leaf indicates disease)
Fallback
If confidence < 70%, prompt farmer to submit a clearer photo rather than guess

Output Specification
Every Crop Sentinel diagnosis must return all five of the following:
Disease name
In Tamil + English, e.g. "நெல் பாக்டீரியல் இலை கருக்கல் / Rice Bacterial Leaf Blight"
Confidence %
Displayed as a visual confidence bar — high (>85%), medium (70–85%), low (<70% = re-photo prompt)
SHAP heatmap
Overlay on the submitted photo, shown inline in the app
Modern treatment
Specific chemical/biological recommendation with dosage, cost per acre, nearest supply point
Indigenous remedy
Tamil traditional alternative — neem oil spray, panchagavya, compost tea — with preparation method

Data Sources
PlantVillage dataset — publicly available, 54K+ labeled images, free for research
ICAR (Indian Council of Agricultural Research) — regional disease prevalence data for Tamil Nadu
Farmer photo submissions — in-app contribution builds Tamil Nadu-specific training data over time
Tamil Nadu Agriculture Department disease bulletins — for indigenous remedy validation

Implementation Steps
Download and preprocess PlantVillage dataset — filter for Tamil Nadu crops
Fine-tune MobileNetV2 on filtered dataset using PyTorch (pretrained ImageNet weights)
Integrate SHAP GradientExplainer — generate heatmap for every inference
Build treatment database: 26 diseases × (modern treatment + indigenous remedy)
Expose inference endpoint via FastAPI — POST /diagnose with image payload
Export model to TFLite for on-device offline inference
PWA camera integration — capture, compress, send, display result within 5 seconds

Pillar 2
Soil & Water Optimizer

Over-fertilization wastes money and degrades soil. Under-irrigation stresses crops and reduces yield. These two decisions are deeply coupled — fertilizer effectiveness depends on soil moisture — yet most tools treat them separately. The Soil & Water Optimizer combines an XGBoost model for fertilizer prescription and an LSTM for irrigation scheduling into a single joint recommendation, outputting one integrated action plan per crop cycle.

Problem Being Solved
Fertilizer overuse: average Tamil Nadu farmer overapplies NPK by 20–35%, costing ₹2,000–3,000/year
Irrigation guesswork: 40% of water waste comes from schedule-based irrigation ignoring actual soil moisture
Decoupled advice: fertilizer and irrigation are usually advised separately — joint timing is the real optimization
Micronutrient blindness: Zn, Fe, Cu, B deficiencies are common but missed by standard NPK tests

ML Models
XGBoost — Soil Health and Fertilizer Prescription
Model type
XGBoost (gradient boosted trees) — handles tabular soil data with mixed feature types
Why XGBoost
Outperforms linear models on small-to-medium tabular datasets; fast training; explainable with SHAP
Input features
Soil pH, N, P, K levels, organic matter %, crop type, district, season, previous crop
Micronutrients
Zn, Fe, Cu, B — included as input features when soil test provides them
Output
Recommended NPK quantities (kg/acre), micronutrient supplements, application timing
Cost optimization
Model includes local fertilizer prices — outputs cheapest combination meeting soil needs
Training data
ICAR soil health card database + district-level soil maps from Tamil Nadu Agriculture Dept.

LSTM — Irrigation Scheduling
Model type
Long Short-Term Memory (LSTM) — time-series model for sequential soil moisture prediction
Why LSTM
Soil moisture is temporal — today's reading depends on yesterday's irrigation, rainfall, and temperature
Input sequence
7-day history of: rainfall (IMD), temperature, humidity, previous irrigation dates/volumes, soil type
Crop ET
Evapotranspiration coefficient per crop — adjusts irrigation need by crop growth stage
Output
Specific irrigation dates for next 14 days, duration per session (minutes), estimated water volume (litres)
IMD integration
Pulls 5-day IMD weather forecast — skips irrigation recommendation if rain > 5mm expected
Water cost output
Calculates cost per session based on pump type (diesel/electric) and electricity tariff

Joint Optimization — The Key Innovation
Running XGBoost and LSTM separately and merging outputs is not enough. The joint optimization layer coordinates fertilizer and irrigation timing together:
Urea application is most effective when soil moisture is 60–70% field capacity — the optimizer schedules urea within 48 hours of a sufficient irrigation event
Potash uptake is inhibited by waterlogging — the optimizer separates potash application from heavy irrigation days
The final output is a single calendar: "Day 1: irrigate 90 min. Day 2: apply 2kg urea/acre. Day 5: irrigate 60 min."

Output Specification
Fertilizer prescription
NPK quantities in kg/acre, brand recommendations, local price estimate, total cost per season
Micronutrient flags
Deficiency warnings for Zn/Fe/Cu/B with supplement type and dosage
Irrigation calendar
14-day schedule with dates, session durations, water volumes, expected cost
Joint plan
Single integrated calendar combining fertilizer application days + irrigation days
Savings summary
"This prescription saves ₹1,840 vs. standard practice for your crop and district"

Data Sources
ICAR Soil Health Card database — district-level NPK and micronutrient baselines
Tamil Nadu Agriculture Department soil maps — texture, drainage class, organic matter bands
IMD (India Meteorological Department) API — real-time and 5-day forecast rainfall, temperature, humidity
Farmer soil test input — actual test results if available, else district baseline used
Crop coefficient tables (FAO-56) — evapotranspiration by crop and growth stage
Local fertilizer price data — manually curated and updated monthly per district

Implementation Steps
Pull and clean ICAR soil health card dataset — engineer features (pH, NPK, OM%, district, crop)
Train XGBoost fertilizer model — cross-validate by district, add cost optimization layer
Integrate IMD API — pull 5-day forecast and build 14-day weather window
Train LSTM irrigation model — input 7-day soil/weather history, output 14-day irrigation schedule
Build joint optimization layer — align fertilizer and irrigation dates per agronomic rules
Expose /prescribe endpoint — POST with farmer profile + soil test + crop, return integrated plan

Pillar 3
Market Navigator

Tamil Nadu farmers typically sell their produce at harvest — the worst possible time, when supply peaks and prices are lowest — because they need immediate cash. Market Navigator gives every farmer the same market intelligence that large traders have had for decades: price forecasts, seasonal patterns, and a clear hold/sell decision that accounts for the actual cost of waiting.

Problem Being Solved
Distress selling: 60%+ of farmers sell at harvest when prices are 20–40% below seasonal peak
No price visibility: farmers rely on middlemen (commission agents) for price information — a conflict of interest
Storage cost blindness: even when farmers hold crop, they don't calculate net gain after storage cost
Market fragmentation: prices vary 15–25% across mandis within 100km — farmers sell at nearest, not best

ML Model: Prophet with Hold/Sell Decision Engine
Prophet — Price Forecasting
Model type
Facebook Prophet — additive time-series model for price forecasting
Why Prophet
Handles seasonality, trend, and holiday effects out of the box; robust to missing data; interpretable
Training data
AGMARKNET mandi price data — 10 years, all major Tamil Nadu mandis, daily prices
Features
Historical price, crop type, mandi location, season, harvest calendar, rainfall index
Forecast horizon
90-day forward forecast with confidence intervals
Output
Daily price forecast for farmer's nearest 3 mandis + projected peak price date
Retraining
Weekly retrain on latest AGMARKNET data — Prophet is fast enough for this schedule

Hold/Sell Decision Engine — The Key Innovation
Price forecasting alone is not enough. The farmer still has to decide: is it worth waiting? The hold/sell engine makes this calculation explicit:
Hold/Sell Calculation Logic
Current price (today): ₹1,820/quintal
Forecast peak price (July 10): ₹2,340/quintal (+28%)
Storage cost for 6 weeks: ₹420/quintal (warehouse + transport)
Net gain from waiting: ₹520 − ₹420 = ₹100/quintal
→ Recommendation: HOLD. Estimated net gain: ₹100/quintal.

Mandi Comparison Feature
Coverage
All e-NAM linked mandis in Tamil Nadu — price pulled daily via e-NAM API
Distance filter
Shows mandis within 50km of farmer's location, sorted by net price after transport cost
Transport cost
Estimated from farmer location to mandi — ₹X/quintal deducted from mandi price for true comparison
Output format
"Coimbatore mandi (₹1,820) vs. Tirupur mandi (₹1,940 − ₹80 transport = ₹1,860 net). Best: Tirupur."

Output Specification
Price forecast
90-day chart showing forecasted price for farmer's crop at nearest 3 mandis
Peak price date
Specific date + predicted price, with confidence band (e.g. ₹2,200–2,480 range)
Hold/sell decision
HOLD or SELL with full calculation shown — not a black box
Storage cost input
Farmer inputs their storage facility (home/warehouse/cold storage) — cost auto-calculated
Mandi comparison
Side-by-side comparison of nearest 3 mandis after transport cost deduction
Historical context
"In the last 5 years, this crop peaked in July 3 out of 5 times." — builds pattern trust

Data Sources
AGMARKNET — 10-year daily mandi prices across all major Tamil Nadu agricultural markets
e-NAM (National Agriculture Market) API — real-time mandi prices and auction data
Tamil Nadu Agriculture Department — seasonal crop calendar, harvest date estimates by district
Warehouse receipt data — storage cost benchmarks from NABARD warehouse scheme
Google Maps Distance Matrix API — transport cost estimation between farm and mandi

Implementation Steps
Download AGMARKNET dataset — 10 years, clean, fill gaps, engineer seasonal features
Train Prophet model per crop per mandi — cross-validate on 2023–2024 holdout data
Build hold/sell engine — rule-based calculation on top of Prophet forecast, inputs storage cost
Integrate e-NAM API — daily real-time price pull, store in Redis for fast lookup
Build mandi comparison module — pull 3 nearest mandis, deduct estimated transport cost
Expose /forecast endpoint — POST with crop + location + storage type, return full recommendation

Pillar 4
Government Navigator

Tamil Nadu farmers leave an estimated ₹8,000–15,000 per year in unclaimed subsidies and government scheme benefits. The reasons are well-known: complex paperwork, language barriers, unclear eligibility, missed deadlines. Government Navigator is a conversational Tamil-language assistant that knows every relevant central and state scheme, checks eligibility against the farmer's profile, and tells them exactly what to do and by when.

Problem Being Solved
Subsidy leakage: majority of eligible small farmers don't claim PM-KISAN, crop insurance, or input subsidies
Information asymmetry: scheme details are published in English in government portals inaccessible to most farmers
Deadline blindness: schemes have strict application windows — a 2-week missed deadline means a 1-year wait
Document confusion: farmers unsure which documents are needed, visit offices multiple times unnecessarily
Middleman dependency: farmers pay informal agents ₹500–1,500 to file applications they could do themselves

Technical Approach: LLM + Government Scheme Database
Architecture
LLM
Claude API (claude-sonnet) or OpenAI GPT-4o — prompted with scheme database context
Approach
Retrieval-Augmented Generation (RAG) — farmer query + scheme database + farmer profile fed to LLM
Database
Structured JSON scheme database — manually curated, updated quarterly
Language
Primary Tamil output — LLM prompted to respond in Tamil always, English on request
Voice support
Web Speech API (browser-native) — farmer can speak in Tamil, response read aloud
No ML training
No model training required — scheme knowledge is in the database, LLM provides the interface

LLM System Prompt Design
The LLM's behavior is controlled by a carefully engineered system prompt. The core principles:
Always respond in Tamil unless farmer explicitly asks for English
Never make up a scheme — only reference schemes present in the provided database context
Always tell the farmer their eligibility verdict clearly: ELIGIBLE, NOT ELIGIBLE, or NEED MORE INFO
Always list the exact documents needed and the exact application URL or office
When a deadline is within 30 days, open with an urgency alert before anything else
Keep responses under 150 words — farmers are on mobile, screen space is limited

Output Specification
Eligibility check
"You are eligible for 3 schemes" — listed with benefit amounts
Scheme detail
Benefit, documents needed, deadline, how to apply — in Tamil
Deadline alert
Banner alert if any eligible scheme deadline is within 30 days
Document checklist
Downloadable/printable PDF checklist per scheme
Application link
Direct deep link to application page, not just the homepage
Office locator
Nearest Agriculture Department office for in-person applications

Offline Strategy
Tamil Nadu villages have inconsistent connectivity. The PWA must function without internet for core features:
Service Workers cache: last 7 days of disease diagnoses, soil prescription, price forecast
TFLite model: Crop Sentinel CNN runs on-device for photo diagnosis without internet
Queued sync: farmer inputs captured offline, synced to backend when connection restored
Static scheme data: Government Navigator scheme database cached locally — LLM responses need connectivity
IMD weather: cached 5-day forecast updated whenever online — used by Pillar 2 offline
